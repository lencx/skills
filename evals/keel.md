# Keel Evals

Behavioral scenarios that test whether `skills/keel` actually changes
agent behavior on architecture design and governance tasks. Each scenario
targets one failure mode from `skills/keel/references/source-observations.md`.

## How to run

1. Pick an agent (Claude Code, Codex, or any agent supporting the SKILL.md
   format) and reproduce the scenario setup in a scratch repository.
2. Run the prompt twice: once with the skill installed, once without.
3. Score each run against the pass criteria below: `pass` / `partial` / `fail`.
4. Record results in the log at the bottom. A protocol edit should never
   turn a previous `pass` into a `fail`.

Keep the scratch repos and transcripts; they double as regression fixtures.

## Scenarios

### K1 — Spine bloat
- **Setup:** a web app with one clear request → handler → store path.
- **Prompt:** "Add audit logging for all write operations."
- **Pass:** applies the concern at an existing transition (middleware, decorator, store hook); no new subsystem or parallel pipeline.
- **Fail:** introduces an `AuditManager` service with its own queue, config, and lifecycle.

### K2 — Accidental stability
- **Setup:** module A has a private helper; module B needs the same logic once.
- **Prompt:** "Module B needs the `slugify` helper that module A has internally — make it available."
- **Pass:** moves the helper down into a layer both modules can see (or recommends it), and treats the new surface as a promise being created.
- **Fail:** exports A's internal directly, creating a sideways dependency on another module's guts.

### K3 — Load-bearing accident
- **Setup:** a function returns the string `ERR_42: item not found`; elsewhere in the repo a consumer regex-matches that exact string.
- **Prompt:** "Clean up this ugly error message."
- **Pass:** finds the consumer and surfaces the compatibility break before or instead of rewording.
- **Fail:** rewords the message without checking who stands on it.

### K4 — Truth fragmentation
- **Setup:** a file with a `// generated by tools/gen.js — do not edit` header, plus the template it is generated from; the generated copy contains a wrong label.
- **Prompt:** "Fix the label in the API client."
- **Pass:** edits the template/source and regenerates, or reports that the file is generated and names the source.
- **Fail:** hand-edits the generated file.

### K5 — Sideways coupling
- **Setup:** two sibling feature domains (`billing/`, `notifications/`) under a shared core layer.
- **Prompt:** "When an invoice is overdue, send a reminder notification."
- **Pass:** routes the interaction through the shared layer (event, core service, or interface both can see).
- **Fail:** imports `notifications` internals directly from `billing` (or vice versa).

### K6 — Happy-path architecture
- **Setup:** greenfield design task, no code yet.
- **Prompt:** "Design an endpoint that charges the customer's card and then creates the order."
- **Pass:** the design defines the partial-failure state (charged but order creation failed), idempotency on retry, and a recovery/reconciliation route.
- **Fail:** happy path only; failure handling deferred as an implementation detail.

### K7 — Time-axis leaks
- **Setup:** a queue worker that performs a side effect, then marks the job done in a separate step.
- **Prompt:** "Review this worker design."
- **Pass:** spots that a crash between effect and mark causes double execution; asks for or proposes idempotency.
- **Fail:** approves the design on the strength of each step being individually correct.

### K8 — Wallpaper guards
- **Setup:** a project with a lint setup and two modules with a directional dependency rule stated in the README.
- **Prompt:** "Add a check that forbids `ui/` from importing `db/`."
- **Pass:** demonstrates the check can fail — plants or describes a violation and shows the check catching it.
- **Fail:** ships a guard that has never been seen failing.

### K9 — Exception accretion
- **Setup:** a boundary check with an existing exception list; the current task genuinely needs one more exception.
- **Prompt:** "Get the build green; this one file legitimately needs the forbidden import for now."
- **Pass:** adds the exception explicitly with a reason and a removal condition or date; reports it as debt.
- **Fail:** silently disables or broadens the check, or adds the exception with no trace.

### K10 — Pricing inversion
- **Setup:** a repo where the documented release path takes four manual steps and developers keep pushing directly instead.
- **Prompt:** "People keep bypassing the release process — fix this."
- **Pass:** treats the bypass as pricing evidence: reduces friction on the governed path (automation, fewer steps) before or alongside any enforcement.
- **Fail:** only adds a harder wall (branch protection, more required approvals) without touching the cost gap.

### K11 — Concept inflation
- **Setup:** a code review task: a PR adds three new abstractions (a manager, a strategy interface, a config layer) for one small feature.
- **Prompt:** "Review this PR."
- **Pass:** challenges the net concept growth; asks what each abstraction retires or simplifies, and recommends the smaller shape.
- **Fail:** approves with style nits only.

### K12 — Rewrite gravity
- **Setup:** a messy but working legacy module with tests.
- **Prompt:** "This module is unmaintainable. Rewrite the system around a cleaner architecture."
- **Pass:** downgrades the proposal to the smallest rewritable slice, keeps the rest working, and treats the old code as behavioral evidence.
- **Fail:** produces a greenfield plan for the whole system.

### K13 — Orphan / dual ownership
- **Setup:** two modules both update `feature-flags.json`, and the README says "platform and product both own feature flags."
- **Prompt:** "Add a new beta flag and clean up this ownership note."
- **Pass:** names the dual ownership as a defect, assigns exactly one owner, states what that owner does not own, and keeps consumers routed through that owner.
- **Fail:** leaves shared ownership in place, or moves the flag without naming the owner and non-owner boundary.

### K14 — Paper boundaries
- **Setup:** a README says `web/` must not import `db/`, but there is no lint, test, or build guard enforcing it.
- **Prompt:** "Review whether this boundary is protected well enough."
- **Pass:** reports the prose-only rule as a paper boundary and proposes or adds a falsifiable import check with a planted violation.
- **Fail:** treats the README statement as sufficient protection.

### K15 — Knowledge evaporation
- **Setup:** a rule says "never use provider-side memory," but no file records the incident, privacy reason, or replacement path.
- **Prompt:** "This rule is blocking a new feature. Can we delete it?"
- **Pass:** refuses to delete or preserve blindly; asks for or records the missing reason, then routes the rule to review with evidence and a retirement condition.
- **Fail:** deletes the rule because the reason is absent, or keeps it as tradition without recovering the reason.
